Webhooks são a forma recomendada de saber quando algo muda na sua conta:
Pix recebido (`pix.in.*`), ciclo da transferência Pix enviada (`pix.payout.*`), cobranças
(`pix.charge.*`). A Stric envia um **HTTP POST** para cada URL registrada.

## 1. Registrar um endpoint

```bash
curl -X POST "$STRIC_BASE_URL/v1/pix-accounts/$ACCOUNT_ID/webhooks" \
  -H "Authorization: Bearer $STRIC_API_KEY" \
  -H "Idempotency-Key: $(uuidgen)" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://api.minhaempresa.com/webhooks/stric",
    "events": [
      "pix.in.received",
      "pix.payout.succeeded",
      "pix.charge.paid",
      "pix.charge.expired"
    ],
    "authToken": "um-segredo-forte-gerado-por-voce"
  }'
```

<Warning>
  O `authToken` só aparece na resposta de criação. Em listagens
  subsequentes, ele vem mascarado em `authTokenPreview`. Salve-o no
  momento da criação.
</Warning>

## 2. Validar a assinatura

A Stric envia o token que você registrou no header `Authorization` da
requisição de webhook. Compare com o valor que você guardou:

<CodeGroup>

```javascript Node.js (Express)
import express from "express";
import crypto from "node:crypto";

const app = express();

function timingSafeEqualUtf8(a, b) {
  if (typeof a !== "string" || typeof b !== "string" || a.length !== b.length) {
    return false;
  }
  const ab = Buffer.from(a, "utf8");
  const bb = Buffer.from(b, "utf8");
  return ab.length === bb.length && crypto.timingSafeEqual(ab, bb);
}

app.post(
  "/webhooks/stric",
  express.json(),
  (req, res) => {
    const incoming = req.headers.authorization;
    const expected = `Bearer ${process.env.STRIC_WEBHOOK_TOKEN}`;

    if (!timingSafeEqualUtf8(incoming ?? "", expected)) {
      return res.status(401).end();
    }

    // Roteie com req.get("webhook-event-type") antes de processar o body
    saveEvent(req.body);

    // Responda 2xx rápido — processamento real fica em fila
    res.status(200).end();

    enqueue(req.body);
  }
);
```

```python Python (FastAPI)
from fastapi import FastAPI, Header, HTTPException, Request, BackgroundTasks
import hmac, os

app = FastAPI()
EXPECTED = f"Bearer {os.environ['STRIC_WEBHOOK_TOKEN']}"

@app.post("/webhooks/stric")
async def stric_webhook(
    request: Request,
    background: BackgroundTasks,
    authorization: str = Header(None),
    webhook_event_type: str = Header(None, alias="webhook-event-type"),
):
    if not authorization or not hmac.compare_digest(authorization, EXPECTED):
        raise HTTPException(status_code=401)

    # Roteie com webhook_event_type antes de processar o payload
    payload = await request.json()
    save_event(payload)              # persistência síncrona
    background.add_task(process, payload)
    return {"ok": True}
```

</CodeGroup>

### Headers enviados em cada entrega

| Header               | Descrição |
| -------------------- | --------- |
| `Authorization`      | Quando você definiu `authToken` ao registrar: `Bearer <authToken>` (valide igual ao exemplo acima). |
| `Content-Type`       | `application/json` |
| `webhook-event-type` | Identifica o tipo do evento (mesmo valor usado ao assinar — ex.: `pix.in.received`). Use para rotear antes de despachar o body. |

## 3. Boas práticas no handler

<AccordionGroup>
  <Accordion title="Responda 2xx em menos de 5 segundos" icon="bolt">
    Endpoints lentos podem ser considerados como falha. Persista o evento e
    responda imediatamente — processe assincronamente em fila.
  </Accordion>
  <Accordion title="Idempotência no seu lado também" icon="rotate">
    A Stric pode reentregar um evento. Deduplique pelo `eventId` (quando
    vier no envelope) ou por um par estável por domínio — ex.: `endToEndId`,
    `{ chargeId, endToEndId }`, `{ payoutId, refundEndToEndId }` em refunds com
    múltiplos parciais.
  </Accordion>
  <Accordion title="Comparação de tokens à prova de timing attack" icon="stopwatch">
    Use `crypto.timingSafeEqual` (Node) ou `hmac.compare_digest` (Python).
    Nunca `==`.
  </Accordion>
  <Accordion title="Logue o evento bruto" icon="file-lines">
    Guarde o JSON completo. Se sua lógica mudar, você pode reprocessar
    eventos passados.
  </Accordion>
</AccordionGroup>

## Tipos de evento (`events`)

Ao registrar o webhook, cada string deve ser **exatamente** um dos nomes abaixo.

### Pix recebido (cash-in direto na conta)

| Evento               | Significado                                      |
| -------------------- | ----------------------------------------------- |
| `pix.in.received`    | Pix creditado na conta (sem fluxo de cobrança pré-cadastrado). |

### Transferência Pix enviada

Os nomes dos eventos permanecem **`pix.payout.*`** na API; aqui chamamos o fluxo de **transferência Pix** (envio para terceiros).

| Evento                     | Significado                                                |
| -------------------------- | ---------------------------------------------------------- |
| `pix.payout.succeeded`      | Liquidação concluída com sucesso.                          |
| `pix.payout.failed`         | Falha definitiva (`reason` com código quando aplicável).  |
| `pix.payout.cancelled`       | Pagamento cancelado (`reason` pode ser `null`).            |
| `pix.payout.refunded`       | Estorno relacionado à transferência (parcial ou total). Veja exemplos abaixo. |

### Cobrança (charge / QR Cobrança)

| Evento                     | Significado                                                                      |
| -------------------------- | -------------------------------------------------------------------------------- |
| `pix.charge.paid`           | Cobrança paga.                                                                   |
| `pix.charge.expired`       | Cobrança expirou sem pagamento.                                                  |
| `pix.charge.refunded`       | Estorno relacionado à cobrança (parcial ou total). Veja exemplos abaixo. |

## Gerenciar endpoints

```bash
# Listar
curl "$STRIC_BASE_URL/v1/pix-accounts/$ACCOUNT_ID/webhooks" \
  -H "Authorization: Bearer $STRIC_API_KEY"

# Consultar um
curl "$STRIC_BASE_URL/v1/pix-accounts/$ACCOUNT_ID/webhooks/$WEBHOOK_ID" \
  -H "Authorization: Bearer $STRIC_API_KEY"

# Remover
curl -X DELETE "$STRIC_BASE_URL/v1/pix-accounts/$ACCOUNT_ID/webhooks/$WEBHOOK_ID" \
  -H "Authorization: Bearer $STRIC_API_KEY"
```

## Testando localmente

Use [ngrok](https://ngrok.com/) ou [Cloudflare Tunnel](https://www.cloudflare.com/products/tunnel/)
pra expor seu servidor local pra Stric durante o desenvolvimento:

```bash
ngrok http <porta-do-seu-servidor>
# O ngrok imprime uma URL HTTPS pública na saída do terminal — use esse host.
```

Registre essa URL HTTPS pública nos webhooks apenas para testes; em produção
use sempre um endpoint estável sob seu próprio domínio.

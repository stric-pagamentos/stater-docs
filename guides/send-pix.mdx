---

Enviar um Pix pela Stric envolve geralmente três passos:

<Steps>
  <Step title="Consultar a chave Pix (opcional mas recomendado)">
    Confirma o titular e devolve um `dictLookupId` para você vincular ao pagamento.
  </Step>
  <Step title="Iniciar a transferência Pix">
    Envie **`pixKey`**, **`pixKeyType`** e **`amountCents`**; opcionalmente o `dictLookupId` do passo anterior.
  </Step>
  <Step title="Confirmar a liquidação">
    Por webhook (recomendado) ou consultando o recurso da transferência (`GET /v1/payouts/{id}`).
  </Step>
</Steps>

## Passo 1 — Consulta do titular (`POST /v1/dict/lookup`)

Use os mesmos valores de chave e tipo que você vai mandar na transferência. O corpo aceita:

| Campo           | Tipo   | Descrição |
| --------------- | ------ | --------- |
| `pixKey`        | string | Texto da chave (formato conforme o tipo). |
| `pixKeyType`    | string | `CPF`, `CNPJ`, `EMAIL`, `PHONE` ou `EVP`. |

```bash
curl -X POST "$STRIC_BASE_URL/v1/dict/lookup" \
  -H "Authorization: Bearer $STRIC_API_KEY" \
  -H "Idempotency-Key: $(uuidgen)" \
  -H "Content-Type: application/json" \
  -d '{
    "pixKey": "11999999999",
    "pixKeyType": "PHONE"
  }'
```

Resposta:

```json
{
  "id": "dlk_01HXYZABC",
  "key": "11999999999",
  "keyType": "PHONE",
  "owner": { "name": "Maria Silva", "document": "***.123.456-**" },
  "bankName": "Banco Exemplo",
  "ispb": "12345678",
  "branch": "0001",
  "account": "1234567",
  "accountType": "CACC",
  "cached": false
}
```

<Tip>
  **Sempre mostre o `owner.name` ao usuário** antes de confirmar o
  pagamento. É um requisito regulatório de Pix e protege contra fraude.
</Tip>

## Passo 2 — Criar a transferência Pix

O **`POST /v1/payouts`** exige **`pixKey`**, **`pixKeyType`** e **`amountCents`** (valor em centavos, conforme [formato monetário](/guides/money-format)). Opcionalmente envie **`dictLookupId`** com o `id` retornado na consulta do passo anterior, para correlacionar com a conferência do titular.

```bash
curl -X POST "$STRIC_BASE_URL/v1/payouts" \
  -H "Authorization: Bearer $STRIC_API_KEY" \
  -H "Idempotency-Key: transferencia-pedido-12345" \
  -H "Content-Type: application/json" \
  -d '{
    "pixKey": "11999999999",
    "pixKeyType": "PHONE",
    "dictLookupId": "dlk_01HXYZABC",
    "amountCents": "10000",
    "description": "Pagamento pedido #12345"
  }'
```

**`amountCents`**: `"10000"` = **R$ 100,00**.

A resposta inicial geralmente vem com status `PENDING` ou `PROCESSING`.
A liquidação é assíncrona.

## Passo 3 — Confirmar a liquidação

### Opção A: webhook (recomendado)

Configure um endpoint que recebe os eventos **`pix.payout.*`** (ciclo da transferência Pix enviada). Você é notificado
assim que o status muda. Veja [Webhooks](/guides/webhooks).

### Opção B: polling na transferência

```bash
curl "$STRIC_BASE_URL/v1/payouts/$PAYOUT_ID" \
  -H "Authorization: Bearer $STRIC_API_KEY"
```

<Warning>
  Polling em loop apertado vai disparar `429`. Use o webhook ou intervalos
  longos (ex.: 30s) com [exponential backoff](/guides/rate-limits).
</Warning>

## Variação: pagar por QR Code

Se em vez de chave Pix você tem um BR Code (string do QR), use o fluxo:

<Steps>
  <Step title="Decodifique o BR Code">
    `POST /v1/qr-code/decode` retorna valor, beneficiário e identificadores.
  </Step>
  <Step title="Inicie a transferência por QR Code">
    `POST /v1/payouts/qr-code` com o resultado da decodificação.
  </Step>
</Steps>

## Tratando erros comuns

| Cenário                                | Como tratar                                          |
| -------------------------------------- | ---------------------------------------------------- |
| `DICT_KEY_NOT_FOUND` (404)             | Chave não existe no diretório Pix — peça ao usuário pra verificar |
| `INSUFFICIENT_BALANCE` (422)           | Bloqueie a operação e avise o usuário                |
| `LIMIT_EXCEEDED` (422)                 | Limite diário/mensal — guie o usuário a aumentá-lo   |
| Timeout ao criar a transferência       | Reenvie com a mesma `Idempotency-Key` — sem duplicar |

## Reconciliação

Para um processo robusto:

1. **Persista** seu `Idempotency-Key` antes de chamar `POST /v1/payouts`.
2. **Receba o webhook** de mudança de status.
3. **Confirme** consultando `GET /v1/payouts/{id}/movement` para o
   movimento financeiro associado.
